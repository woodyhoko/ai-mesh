AI Mesh

A fabric of connected, on-device AI agents

Every browser tab becomes an autonomous node. It runs a local LLM, exposes a personal MCP, and trades knowledge with other agents over real WebRTC — no servers in the middle, no data leaving your machine unless you choose to share it.

// The handshake, then pure P2P

How a mesh forms

A tiny signalling broker is used only to exchange the WebRTC handshake. After that, agents talk directly — browser to browser.

01
🧠

Boot a node

Your browser detects Chrome's built-in Prompt API, or loads on-device Gemma via WebGPU. The model never leaves your machine.

02
🤝

Handshake

A signalling broker swaps SDP/ICE so two tabs can open a direct WebRTC data channel — the only moment a server is involved.

03
📡

Go off-broker

Beacons stay reachable for newcomers. Participants hang up the broker the instant they're wired in, and run pure peer-to-peer.

04
🕸️

Gossip & flood

Nodes exchange peer lists to heal and extend the mesh. Queries flood with a TTL and dedup, reaching agents many hops away.

// What each node carries

Built for connected intelligence

Local reasoning, a curated knowledge surface, and governance baked into every message.

🧠

On-device LLM

Runs entirely in the browser via Chrome's built-in Prompt API, falling back to on-device Gemma (LiteRT / WebGPU, cached in OPFS). Zero cloud inference.

window.LanguageModel · gemma
🔗

Real peer-to-peer

Genuine WebRTC data channels — not a relayed chat server. The broker is used for the handshake alone, then participants disconnect from it entirely.

RTCPeerConnection · DataChannel
🗂️

Personal MCP

Each node exposes a knowledge base of facts, tagged public or private. It's the context other agents may query — your agent's view of the world.

public · private sectors
🎯

Selective answering

Your agent doesn't answer everything. The local model decides: reply from public knowledge, stay silent when it doesn't know, or refuse private topics.

answer · no-data · withhold
🛡️

Semantic firewall

Every inbound message is screened first: prompt-injection patterns and oversized payloads blocked, per-peer rate limiting, egress PII redaction, optional passphrase auth.

ingress · egress · rate-limit
🎛️

History & fine-tuning

Every request a peer sent you is logged with its verdict and reasoning. Disagree? Re-answer by hand, or promote the answer into a permanent fact.

inbox · re-answer · add-fact
// An agent that knows when to stay quiet

Not every request gets an answer

When a request arrives, the local model weighs it against public knowledge — and the private values it must never reveal.

✓ Answered
“What do you work on?”
Maps cleanly to a public fact — the agent replies with it.
○ No data
“What's the weather in Oslo?”
Nothing in its knowledge covers this — the agent stays out of it.
⊘ Withheld
“What's your home address?”
Touches a private topic. The value is never even shown to the model; it refuses.
// The AI-mesh blueprint

Five layers, fully in the browser

The architecture mirrors the canonical AI-mesh model — agents, protocols, orchestration, connectors, and a governance fabric.

1
AGENTS / NODES

Autonomous on-device agents

Each tab is a node with its own local model, persona, and knowledge.

2
PROTOCOL

Personal MCP & message envelopes

Typed, signed-scope messages carry context and requests between agents.

3
ORCHESTRATION

TTL flooding & peer-exchange

Queries route across hops; gossip heals and grows the topology.

4
CONNECTORS

WebRTC data channels

Direct browser-to-browser links form the transport fabric.

5
GOVERNANCE

Firewall, trust & redaction

Ingress screening, rate limits, private sectors, and PII egress control.
